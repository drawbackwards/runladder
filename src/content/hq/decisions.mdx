## How to use this page

The why behind big choices. Read this before you push back on a feature, a price, or a roadmap call. Most decisions look arbitrary until you know the constraint that drove them. New entries go at the top with a date and one-line rule.

If you want to challenge a decision, file an issue in GitHub with the proposed change and link the entry. We update entries when the world changes, never silently.

## Site password gate bypasses signed-in users (2026-05-26)

**The runladder.com password wall (`SITE_PASSWORD`) is bypassed for any signed-in user, including invitees returning from Clerk's hosted portal with an active session. Cold visitors still hit the wall.**

Why: the gate (`PasswordGate.tsx`) wraps the whole app and blocked the Team onboarding flow. An invited customer would accept a Clerk invite, return to runladder.com, and hit the password wall before reaching `/dashboard`. Bypassing signed-in users keeps the gate effective against casual visitors while letting real customers onboard. The decision is reversible and low-risk: the entire gate is dead code once `SITE_PASSWORD` is removed from Vercel, since the layout never renders it when the env var is unset.

How to apply: do not treat the password gate as a security boundary. It is a client-side speed bump, and `/api/*` routes are not behind it. Real access control is Clerk auth plus the admin/team allowlists. Remove the gate (`PasswordGate.tsx`, `/api/auth-gate`, the layout conditional) once marketing edits land and `SITE_PASSWORD` is dropped. Shipped in PR #198.

## Post-merge /hq verification across all Ladder platform repos (2026-05-19)

**UNBREAKABLE rule.** Every time a PR ships live in any of the four Ladder platform repos (`runladder`, `ai-design-assistant`, `ladder-beta`, `ladder`), Sara verifies `/hq` is current and prints a confirmation to Ward.

The confirmation shape:

> **HQ verified for PR #N (vX.Y.Z).**
> - Pages updated: list (or "none, all current")
> - Pages spot-checked: list
> - Live at runladder.com/hq

Silence after a deploy is not acceptable. Even if no `/hq` update was needed, say so explicitly. The handshake is the discipline.

Why: the v0.5.1 audit revealed `/hq` had drifted because doc updates were happening reactively. The post-merge verification print forces the check to happen at the natural moment (just shipped) and makes the discipline visible to Ward.

How to apply:

- After every Ladder-platform PR merge, walk the `/hq` pages most likely affected (Features, Journeys, API, Decisions, Roles, Roadmap, Architecture, Glossary, Brand, GTM).
- If updates are needed, open a follow-up PR. Bump `lastPr` per [the lockstep protocol](/hq/decisions#hq-stays-in-lockstep-with-the-code).
- If no updates needed, print the confirmation with "Pages updated: none, all current."
- Documentation-only PRs to non-platform repos (engineering-standards, drawbackwards-knowledge, etc.) do not trigger this rule.

This rule sits alongside [the lockstep protocol](/hq/decisions#hq-stays-in-lockstep-with-the-code) and [the public-pricing rule](/hq/decisions#hq-documents-public-pricing-only). All three are part of the same hygiene system.

How to apply:

- Ward and Michael keep the full pricing context in non-/hq memory. Ask if you need a number for a sales conversation.

## Every claim of "live" in /hq must point to a route, file, or PR (2026-05-19)

**UNBREAKABLE rule.** A feature is tagged Live only if you can name the file path, route, or PR that proves it. Otherwise tag it In flight (code exists, partial) or Roadmap (no code yet).

Why: the v0.5.0 / v0.5.1 seed of `/hq` invented a few things that didn't exist (an `api.runladder.com` domain that wasn't registered, a `/dashboard/settings` route that doesn't exist, an MCP server with three named tools). Ward caught it in the v0.5.2 audit. The fix is a discipline: no Live without evidence. The fix is also a workflow: I do not write a feature row claiming Live without first grepping the codebase.

How to apply:

- Every row in [/hq/features](/hq/features) Live column must have an Evidence column pointing to a file or PR.
- Every flow in [/hq/journeys](/hq/journeys) tagged Live must reference a real route handler.
- Every endpoint in [/hq/api](/hq/api) Live table must exist under `src/app/api/`.
- If the answer to "does this exist" is "kind of" or "partially", it's In flight, not Live.
- If the answer is "we plan to", it's Roadmap.

## /hq stays in lockstep with the code (2026-05-19)

**UNBREAKABLE rule.** Every PR that touches code in this repo updates the matching `/hq` page in the same PR. Bump `updatedAt`, `updatedBy`, and `lastPr` in the page's frontmatter. The `LastUpdated` header renders the stamp so anyone reading the page can see when it was last verified against the code and which PR did the verifying.

Why: `/hq` is the team's source of truth. If it drifts from the code, it dies. The only way to keep it alive is to make doc updates part of the same commit as code changes. Ward made this an unbreakable rule when we stood `/hq` up.

How to apply: see the project [CLAUDE.md](https://github.com/drawbackwards/runladder/blob/main/CLAUDE.md) for the map of "when you change X, update Y in `/hq`". If a code change has no doc consequence, say so in the commit body. Otherwise the doc update is part of the PR.

| When you change... | Update... |
| --- | --- |
| A user-visible web/plugin/Skill feature | `/hq/features` row, `/hq/journeys` flow if it changes |
| An API endpoint, MCP tool, or auth shape | `/hq/api` |
| Pricing, positioning, roles, model, ICP, or any strategic decision | `/hq/decisions` |
| `src/lib/admin.ts` auth tiers | `/hq/roles` |

## Positioning

**Ladder is agent-native measurement infrastructure, not SaaS-with-AI.**

Why: the market is full of design QA tools and accessibility scanners. None of them give an AI agent a number it can act on. Ladder's wedge is that every surface (web, plugin, Skill, MCP, API) is callable by a human or an agent, returns the same score, and that score is the contract.

How to apply: when in doubt between "make this prettier for humans" and "make this callable by an agent," favor the agent path. Humans get UI on top of the same API.

## Never expose prompts, rubric, or scoring logic

**UNBREAKABLE rule.** Prompts, rubric, scoring instructions, model parameters, and any internal reasoning stays server-side. Every surface is a thin client over the API.

Why: the rubric is the moat. The scoring engine is what makes a Ladder score worth trusting. If the prompt leaks, anyone can replicate the framework and our pricing collapses to zero.

How to apply: never log a prompt to a client surface. Never serialize the rubric into Figma plugin code, the Skill bundle, or any browser-visible payload. The `scripts/check-no-prompt-leak.mjs` pre-build check exists for exactly this reason. If you trip it, fix the leak before you fix the lint warning.

## Public launch is gated on MCP, Skill, and API being live

**No public marketing push until all three developer surfaces ship.**

Why: positioning is agent-native. Launching the consumer site before the agent infrastructure makes us look like another design tool. Launching after positions us as infrastructure with a polished consumer face.

How to apply: ship MCP, Skill, and v1 of the public API before any paid acquisition spin-up. Soft launches with the design community on the framework page are fine.

## Public pricing: Free 5-lifetime, Pro $1,000/mo self-serve, Team and Pulse Custom

**Free** is a try-before-you-buy at 5 scores per account, ever. **Pro** is $1,000/mo for individuals, self-serve on Stripe. **Team** is publicly Custom (Apply to be a beta tester CTA on `/pricing`). **Pulse** is publicly Custom (Contact CTA).

Public limits (visible on `/pricing` and enforced in `src/lib/plans.ts`):

| Tier | Limit | Surfaces |
| --- | --- | --- |
| Free | 5 lifetime scores | Web + Skill + Figma |
| Pro | 2,000 scores/mo (soft cap, hard cap 2x) | All surfaces + API keys (when public API ships) |
| Team | 25,000 pooled scores/mo (Beta) | All Pro features + manager workflow |
| Pulse | Custom (set per engagement) | All Team features + Pulse data ingest in `ladder-beta` |

Pro went from $250/mo to $1,000/mo in v0.4.8 ([PR #166](https://github.com/drawbackwards/runladder/pull/166)) on 2026-04-28. Subscribe button is live via Stripe.

Why: the Free to Pro gap was a leaky funnel when Free was generous. Lifetime cap forces the upgrade decision. Team stays contact-only because the buyer is a Director of Design, not a credit-card buyer, and the deal value supports a sales motion.

How to apply: don't offer ad-hoc discounts on Pro. Don't offer self-serve Team. Internal price targets stay in memory per [/hq documents PUBLIC pricing only](#hq-documents-public-pricing-only).

## Teams is an agency engagement, not self-serve SaaS

**Teams is sold as an MSA + SOW Drawbackwards engagement.** Up to 10 seats included in the SOW. Stripe is used only for usage overage past the included pool. Renewal is annual via SOW, not auto-billed subscription.

Why: Teams buyers are companies with design quality problems, not seat-buyers. The deal includes humans (workshop, audit, onboarding) that don't fit a SaaS contract. Selling Teams as SaaS undervalues the engagement and starves the agency.

How to apply: any "team plan signup" copy on runladder.com routes to a contact form, not Stripe checkout. Sales runs through Ward + Michael.

## ICP: founder-built app stuck at Functional

**Primary ICP archetype.** Founder built the product. App scored 1.5–2.5 on Ladder. Internal growth team was hired and is fighting design debt. No design quality infrastructure in place.

Why: this archetype has the sharpest pain (the founder can see the score, the team can see the score, no one has a system to climb the rungs), the clearest budget (growth team has a number to hit), and the cleanest path to Lift → Teams.

How to apply: marketing copy speaks to this archetype. Top 100 entries that match the archetype get teardown priority. Skip enterprise design system buyers for now.

## Ladder level order is always 5 → 1

**Always render rungs high to low.** Meaningful first, Functional last. Never low to high. Applies to UI, copy, docs, emails, slides, sales decks, everywhere.

Why: the framework is aspirational, not a checklist. Starting at Meaningful sets the bar. Starting at Functional anchors to the floor and reads as a defect ladder, not a quality framework.

How to apply: if you see 1 → 5 anywhere, file an issue. No exceptions.

## One free anonymous score on web; sign-up prompted after the result (revised #187)

**A signed-out visitor can score ONE screen on the web, see the result, then gets the sign-up prompt.** This reverses the v0.4.14 "no anonymous scoring on any surface" decision — for the **web surface only**. Skill, Plugin, MCP, and API still gate at the door.

Why the reversal: the hard gate blocked the top-of-funnel demo. Ward needs to send a prospect straight to the scoring tool and let them feel the product before committing. One free score does that while keeping the conversion lever.

Guarding the original cost concern: anonymous scoring is bounded by a per-browser cap (`ANON_LIMIT`, keyed on the `ladder_anon_id` cookie) plus a higher per-IP daily backstop (`rate:anon:{ip}`). Usage is only counted after a successful score. The scored result is stashed (`pending-score:{anonId}`) and **claimed onto the account on sign-up**, so attribution is preserved — the anon score becomes the user's first counted lifetime score.

How to apply: the `/score` page lets anonymous visitors score once; the post-score CTA and the limit-reached prompt drive sign-up. The free-tier ceiling (5 lifetime) still applies once they have an account.

A11y audit and UX copywriting are **Pro-only**, not free. Free gets the base Ladder score.

## Trademark and proprietary framework

**The Ladder framework is proprietary.** Trademark filings and legal protection are in motion. Treat the rubric, scoring math, and level definitions as IP.

Why: a public framework that anyone can copy collapses the moat. Ladder needs to be cite-able by name and defensible in court.

How to apply: framework page on runladder.com describes the framework. The full rubric does not appear anywhere a competitor can scrape it. Public references must include the trademark symbol once it's filed.
